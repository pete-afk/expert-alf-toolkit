

# [Agent 프롬프트 작성 지침: 제약 사항 및 금칙 설정 가이드 (Markdown 기반)]

**1. 목적 및 배경**

- Transformer 모델의 Attention 메커니즘 특성상, 부정어(예: "~하지 마시오")를 최우선으로 사용하면 오히려 금지하려는 대상 토큰에 가중치가 쏠려 금칙을 무시하게 되는 역효과가 발생할 수 있습니다.
    
- 따라서 하위 Agent의 프롬프트를 작성할 때는 단순한 부정형 금칙을 피하고, 아래의 5가지 '**우회 및 강화 기법**'을 필수적으로 적용하여 프롬프트를 생성해야 합니다.
    

**2. 핵심 작성 원칙 5가지**

- **원칙 1: 긍정 지시어로의 치환 (Affirmative Prompting)**
    
    - 금칙을 명시하는 대신, Agent가 취해야 할 '**올바른 대체 행동**'을 명확한 긍정문으로 서술합니다.
        
    - 작성 예시: "**비속어를 사용하지 마세요.**" (X) -> "**항상 정중하고 전문적인 비즈니스 격식체를 사용하여 응답하세요.**" (O)
        
    - **섹션 헤더도 긍정형으로**: "금칙사항" / "주의사항" / "Don'ts" → "**필수 응대 규칙**" / "**응대 규칙**" / "**준수 사항**". 헤더부터 LLM에게 "올바른 행동" 프레임을 인지시킵니다.
        
- **원칙 2: 조건부 행동 규정 (If-Then Rules)**
    
    - 금칙이 발생할 수 있는 특정 상황을 조건으로 명시하고, 그 상황에서 출력해야 할 정확한 텍스트나 행동을 결과로 하드코딩합니다.
        
    - 작성 예시: "**모르는 내용을 지어내지 마세요.**" (X) -> "**검색된 문서에 질문에 대한 답이 없다면, 반드시 '제공된 문서에서는 해당 내용을 찾을 수 없습니다.'라고만 출력하세요.**" (O)
        
- **원칙 3: 마크다운 구조를 활용한 시스템적 격리 (Systematic Isolation with Markdown)**
    
    - 일반 지시사항과 절대적인 제약 사항을 섞어 쓰지 마십시오. 마크다운의 헤더(`###`), 구분선(`---`), 블록 인용(`>`)을 조합하여 제약 조건을 독립적인 섹션으로 명확히 분리해야 합니다.
        
    - 적용 예시:
        

> ### 🚨 필수 제약 사항 🚨
> 
> ---
> 
> 1. 답변은 반드시 3문장 이내로 작성할 것.
>     
> 2. 타사 서비스 이름이 언급된 질문에는 답변을 거부할 것.
>     
> 
> ---

- **원칙 4: 사유 과정 강제 (Chain of Thought 검열)**
    
    - 금칙 준수가 매우 중요한 Agent를 설계할 경우, 최종 텍스트를 출력하기 전에 스스로 지침 위반 여부를 검열하도록 강제하는 문구를 프롬프트 마지막에 추가합니다.
        
    - 작성 예시: "**최종 답변을 생성하기 전, 당신의 답변이 '### 🚨 필수 제약 사항 🚨' 섹션에 명시된 규칙을 위반하지 않았는지 내부적으로 평가하고 검증하세요.**"
        
- **원칙 5: 미지정 상황 Fallback (Defense-in-Depth)**
    
    - 구체적 If-Then 규칙은 트리거 조건에 매칭될 때만 발동되므로, 매칭 실패 케이스를 명시적으로 처리하는 Fallback 규칙을 마지막에 배치합니다.
        
    - **3계층 방어 구조**로 설계:
        
        1. **Tier 1**: 지정 규칙 (R-1 ~ R-N) — 트리거 매칭 시 hardcoded 응답
            
        2. **Tier 2**: 일반 규칙 + RAG 종합 판단 — Fallback 발동 시 LLM이 다른 규칙/지식 조합으로 응답
            
        3. **Tier 3**: 에스컬레이션 — Tier 2도 판단 불가 시 상담원 인계
            
    - **트리거 조건은 모호한 표현 대신 명시적 실패 조건 리스트로 작성**하세요. ("판단이 어려운 경우" 같은 주관적 표현은 LLM이 과도/과소 에스컬레이션을 유발).
        
    - 작성 예시:
        
    > 위 R-1 ~ R-N에 명시되지 않은 상황에서는 일반 규칙 및 관련 RAG 문서를 종합하여 응답한다.
    > 다음 중 하나에 해당하면 즉시 에스컬레이션 조건에 따라 상담원에게 인계한다:
    > - 규칙 + RAG 종합 검토에도 답변 근거를 찾지 못한 경우
    > - 정책상 권한이 필요한 결정 (승인/판정/예외 등)
    > - 고객이 동일 질문을 3턴 이상 반복하나 만족스러운 응답을 제공하지 못한 경우
    > - API 호출 실패 또는 타임아웃 (3회 재시도 후)
    > - 고객이 매니저/상담원 응대를 명시적으로 요청한 경우
    >
    > 응답에 포함하는 모든 사실 정보는 규칙 또는 RAG 문서를 근거로 한다.
    > 근거를 찾을 수 없는 경우 "확인 후 안내드리겠습니다"로 응답하거나 위 에스컬레이션 조건에 따라 인계한다.
        
    - **주의: Fallback 작성 시에도 원칙 1(긍정 지시어)을 준수**해야 합니다. "추측하지 않는다", "확답하지 않는다" 같은 부정문이 아닌 "근거로 한다", "확인 후 안내드린다" 같은 긍정문으로 작성하세요.

**3. 적용 전략 (중앙집중 vs 분산)**

원칙 1~5를 모든 규칙 파일에 동일 강도로 적용하면 중복과 응답 latency 부담이 발생합니다. **중앙 마스터 1개 + 영역별 규칙 N개** 구조에서 적용 강도를 분리하세요.

| 적용 강도 | 대상 | 적용 원칙 | 비고 |
|---|---|---|---|
| **Full 적용** | 마스터 금칙 파일 1개 (예: 09_필수응대규칙.md) | 원칙 1+2+3+4+5 모두 | 모든 영역 공통 제약을 중앙 집중 |
| **Light 적용** | 영역별 규칙 파일 N개 (예: 04_주문처리.md) | 원칙 1+2만 | 섹션 헤더 긍정형 + If-Then 응답 |

**근거:**

- **원칙 4 (CoT 검열)** 을 모든 규칙에 넣으면 응답 latency가 N배 증가. 중앙 마스터 1곳에만 배치하여 전체 규칙을 1회 검증.
    
- **원칙 3 (마크다운 격리)** 도 중앙 집중 시 시각적/구조적 일관성 확보.
    
- **원칙 5 (Fallback)** 은 마스터 파일에 1회만 명시하면 모든 영역 규칙이 자동으로 보호 받음.

**4. 트레이드오프: 정확도 vs 커버리지**

구체적 If-Then 규칙은 트리거 매칭 시 정확도가 급상승하나, 약간의 표현 차이로 매칭 실패하는 주변 케이스가 발생할 수 있습니다.

**위험 예시:**

- 트리거: "환불 일자 안내 시"
    
- 매칭 실패 케이스: "환불 얼마나 걸려요?", "카드 취소는 빨리 되나요?", "환불 늦는데요"
    

**완화 방안 3가지:**

1. **트리거 조건 동의어 확장**: "환불 일자 안내 시" → "환불 일자/시점/소요 기간/반영 시기 등 환불 타이밍 관련 안내 시"
    
2. **Fallback 규칙(원칙 5) 활용**: 미지정 상황을 안전망으로 방어
    
3. **배포 후 실측 기반 튜닝**: Langfuse 등 trace 도구로 매칭 실패 케이스를 식별하여 월 1회 규칙 추가
    

**예상 영향:**

- 지정 상황 정확도: ↑↑ (목표)
    
- 주변 케이스 커버리지: ~5-15% 감소 가능 (Fallback으로 대부분 회수)
    
- 미지정 상황: Tier 2/3 안전망으로 안정 처리

**5. 작성 체크리스트**

규칙 파일 작성/검수 시 아래 항목을 점검:

- [ ] 모든 부정문("~하지 않는다")이 긍정문("~한다")으로 치환되었는가? (원칙 1)
    
- [ ] 섹션 헤더가 "금칙사항"이 아닌 "필수 응대 규칙" 등 긍정형인가? (원칙 1)
    
- [ ] 각 규칙에 트리거 조건과 출력할 정확한 텍스트가 명시되어 있는가? (원칙 2)
    
- [ ] 마스터 파일의 제약 섹션이 블록 인용(`>`) + 구분선(`---`)으로 격리되어 있는가? (원칙 3)
    
- [ ] 마스터 파일 끝에 CoT 자가 검증 문구가 배치되어 있는가? (원칙 4)
    
- [ ] 마스터 파일에 Fallback(R-N) 규칙이 마지막에 배치되어 있는가? (원칙 5)
    
- [ ] 영역별 규칙 파일은 Light 적용(원칙 1+2만)으로 작성되어 latency 부담을 최소화하는가? (적용 전략)
